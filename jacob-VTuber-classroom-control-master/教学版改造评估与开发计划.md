# Open-LLM-VTuber 教学版改造评估与开发计划

更新时间：2026-02-10

## 1. 目标与结论

你希望把当前项目改造成“面向学生、可教学拆解、低门槛使用”的版本。基于现有代码，7 项需求总体都**可以实现**，但复杂度差异较大：

- 快捷启动、背景上传、人设编辑、Live2D 上传、X-ray 教学入口：可实现，难度中低到中高。
- 音色与语速可视化配置：可实现，但需要补齐前后端能力（当前 TTS 设置页基本为空）。
- 知识库管理：**原项目没有可直接使用的完整知识库模块**，可从零实现为教学版的核心扩展，难度最高。

建议按“先易用性、后内容管理、再知识库”的顺序迭代。

## 4. 面向教学版的总体技术方案

## 4.1 先决条件（必须先做）

1. **前端开发入口切换**
   - 不建议直接修改 `frontend/assets/main-*.js`（编译产物，不可维护）。
   - 建议在 `frontend` 子模块切到源码分支（如 `origin/main` 或 `origin/dev`），在子模块里开发并重新构建。

2. **新增“教学版后端 API 层”**
   - 在现有 `routes.py` 基础上新增 `edu_*` 系列 REST API（上传、设置、知识库、源码浏览）。
   - WebSocket 保持兼容，仅补必要新消息类型。

3. **配置持久化策略**
   - 保持现有 `conf.yaml + characters/*.yaml` 机制。
   - 教学版运行时编辑优先写入 `characters/` 下新文件，避免污染基础配置。

## 5. 分需求详细设计与实现步骤

## 5.1 需求 1：桌面双击启动 + 自动打开网页 + 自动全屏

### 设计要点

- 启动脚本负责：
  1) 激活 venv
  2) 启动后端
  3) 等待端口 ready
  4) 调起浏览器并全屏打开 `http://127.0.0.1:12393`
- 全屏不建议依赖前端 JS `requestFullscreen`（受用户手势限制）。
- 推荐浏览器参数（按系统）：
  - Chrome/Edge：`--new-window --start-fullscreen`
  - 课堂锁屏模式可选：`--kiosk`

### 交付物

- `scripts/start_teaching_mode.sh`（Linux/macOS）
- `scripts/start_teaching_mode.bat`（Windows）
- `scripts/stop_teaching_mode.*`
- 桌面快捷方式说明文档（或自动生成脚本）

### 关键步骤

1. 编写启动脚本：拉起 `./venv/bin/python run_server.py`。
2. 轮询 `http://127.0.0.1:12393/` 直到返回 200。
3. 调起浏览器全屏。
4. 后台日志落盘（如 `logs/teaching_mode.log`）。

### 验收标准

- 学生双击图标后 10~20 秒内自动打开全屏页面。
- 失败时给明确弹窗/日志（端口占用、依赖缺失等）。

## 5.2 需求 2：网页可选音色 + 调语速

### 现状差距

- 前端 TTS 设置页为空：`frontend(origin/main):.../tts.tsx:1-7`。
- 后端当前 `edge_tts` 仅支持 `voice`，无 `rate`：`src/open_llm_vtuber/config_manager/tts.py:150`、`src/open_llm_vtuber/tts/edge_tts.py:17`。

### 设计要点

- 教学版优先支持 `edge_tts`（可在线获取大量语音）。
- 新增统一接口：
  - `GET /edu/tts/voices?engine=edge_tts`
  - `POST /edu/tts/apply`（voice、rate/speed）
- 运行时生效：更新 `ServiceContext.character_config.tts_config` 后执行 `init_tts()`。

### 后端改造步骤

1. 扩展 `EdgeTTSConfig` 增加 `rate` 字段。
2. 扩展 `tts/edge_tts.py`，调用 `edge_tts.Communicate(text, voice, rate=...)`。
3. 扩展 `tts_factory.py` 将 `rate` 透传。
4. 新增 API：列语音、应用参数、（可选）试听。

### 前端改造步骤

1. 实现 `TTS` 设置页组件：音色下拉、语速滑杆、试听按钮。
2. Save 时调用 `/edu/tts/apply`。
3. 同步 UI 状态与当前配置。

### 验收标准

- 切换音色后下一句生效。
- 调整语速后可听出明显变化。
- 保存后重启仍保留（写入角色配置）。

## 5.3 需求 3：背景上传并设为当前背景

### 现状支持

- 背景目录已静态挂载：`src/open_llm_vtuber/server.py:126`。
- 背景列表已有 WebSocket 拉取：`src/open_llm_vtuber/websocket_handler.py:550`。

### 设计要点

- 新增上传接口：`POST /edu/backgrounds/upload`。
- 上传后写入 `backgrounds/`，然后返回可访问 URL（`/bg/<filename>`）。
- 前端上传成功后：
  1) 刷新背景列表
  2) 立即设置当前背景

### 后端关键安全点

- 扩展名白名单：`.jpg/.jpeg/.png/.webp`
- 大小限制（建议 5~10MB）
- 随机化文件名 + 路径净化（防止目录穿越）
- MIME 与内容双校验

### 验收标准

- 上传后立即可在下拉列表看到并切换。
- 非图片格式、过大文件被拦截并提示。

## 5.4 需求 4：结构化人设编辑（性格/性别等）并生效

### 设计要点

- 新增“人设编辑器”而非只给纯文本：
  - 性格标签（外向/严谨/幽默...）
  - 性别/称呼
  - 说话风格
  - 教学目标与禁忌
  - 可选“自由补充”
- 提交时生成 `persona_prompt` 模板文本，写入角色配置并重建 agent。

### 运行时生效路径

- 更新 `character_config.persona_prompt`
- 调用 `construct_system_prompt` + `init_agent`（见 `src/open_llm_vtuber/service_context.py:465` 与 `src/open_llm_vtuber/service_context.py:472` 附近逻辑）

### 推荐数据结构

- 新增：`characters/persona_presets/*.yaml`
- 前端表单 -> 后端模板渲染 -> 生成最终 prompt

### 验收标准

- 点击“应用”后新对话立即体现新 persona。
- 可导出/保存为新的角色 preset，学生可复用。

## 5.5 需求 5：知识库管理模块（上传文档）

### 现状结论

- 当前项目没有可直接上线的知识库模块；`mem0` 相关实现未落地（空文件）。
- 因此该功能需要**新增完整链路**。

### 教学版建议（分阶段）

#### Phase A（先可用）

- 支持上传文档：`pdf/docx/txt/md`
- 解析 -> 分块 -> 向量化 -> 本地向量库（如 FAISS/Chroma）
- 在问答时检索 Top-K 片段，拼接到提示词中

#### Phase B（再增强）

- 多知识库空间（按课程/班级）
- 文档版本、重建索引、命中可视化
- 引用来源高亮（用于教学讲解）

### 最小 API 设计

- `POST /edu/kb/upload`
- `GET /edu/kb/list`
- `POST /edu/kb/reindex`
- `DELETE /edu/kb/:doc_id`
- `POST /edu/kb/select`（切换当前知识库）

### 与现有对话链路集成

- 在 `handle_conversation_trigger` 进入 agent 之前，按当前会话问题做检索。
- 将检索结果追加为系统上下文或工具结果，确保可解释。

### 验收标准

- 上传文档后可被问答命中。
- 返回中可看到“引用来源”。

## 5.6 需求 6：Live2D 形象上传与设定

### 现状约束

- `/live2d-models/info` 能扫描目录，但实际模型切换依赖 `model_dict.json` + `live2d_model_name`。
- 若 `model_dict.json` 未同步，切换会报错（`src/open_llm_vtuber/live2d_model.py:134`）。

### 设计要点

- 上传方式：`zip`（建议）或目录拖拽。
- 上传后自动校验：
  - 结构是否包含 `xxx.model3.json`
  - 资源文件是否完整
- 自动写入或更新 `model_dict.json`。
- 自动生成角色配置模板（写入 `characters/xxx.yaml`），供前端切换。

### API 设计

- `POST /edu/live2d/upload`
- `GET /edu/live2d/list`
- `POST /edu/live2d/select`

### 验收标准

- 上传模型后无需手工改 JSON/YAML 即可在界面选择并生效。

## 5.7 需求 7：X-ray 图标打开对应源代码

### 目标

- 每个模块右侧增加“教学入口”，学生可看到该功能对应源码。

### 推荐实现（安全可控）

- 前端每个模块加 X-ray 图标。
- 点击后打开“源码讲解页”（新面板/抽屉），通过后端 API 拉取源码文本：
  - `GET /edu/source?module=tts`
- 后端维护白名单映射（模块 -> 可读文件列表），禁止任意路径读取。

### 可选增强

- 增加“跳转 GitHub 对应文件行号”按钮。
- 增加“关键函数流程图”。

### 验收标准

- 任一设置模块都能一键看到相关源码和解释。

## 6. 建议开发排期（单人）

## Sprint 0（1~2 天）基础准备

1. 前端切到可维护源码分支并能本地构建。
2. 建立 `teaching` 分支与 CI 检查。
3. 明确教学版配置目录结构。

## Sprint 1（2~3 天）易用性启动

1. 脚本启动 + 自动打开 + 全屏。
2. 错误提示与日志。
3. 桌面快捷方式模板。

## Sprint 2（4~5 天）可视化资源管理

1. 背景上传 + 选择。
2. Live2D 上传 + 注册 + 切换。
3. 前端设置页联调。

## Sprint 3（4~5 天）人设与语音教学化

1. TTS 页面（音色/语速/试听）。
2. 结构化人设编辑器与运行时生效。
3. 保存为角色预设。

## Sprint 4（7~10 天）知识库 MVP

1. 文档上传、解析、索引。
2. 检索增强问答（RAG）。
3. 知识库管理 UI（列表、删除、重建）。

## Sprint 5（2~3 天）X-ray 教学入口

1. 模块级源码映射。
2. 源码查看页与注释。
3. 课堂演示脚本与验收。

总计：约 20~28 天（单人），可拆成 4~6 周教学项目。

## 8. 风险与规避

1. **前端改造入口错误风险**
   - 风险：直接改打包文件导致不可维护。
   - 规避：必须在 `Open-LLM-VTuber-Web` 源码分支开发。

2. **上传安全风险**
   - 风险：任意文件写入、路径穿越。
   - 规避：扩展名白名单 + MIME 校验 + 文件名重写 + 目录隔离。

3. **知识库性能风险**
   - 风险：大文档索引慢、内存高。
   - 规避：异步索引队列 + 文档大小限制 + 可中断重建。

4. **自动全屏兼容性风险**
   - 风险：不同浏览器策略差异。
   - 规避：优先浏览器启动参数，不依赖前端 JS 自动全屏。

## 9. 我建议你先确认的 6 个产品决策

1. 教学版优先支持平台：仅 Windows，还是 Windows + macOS + Linux？
2. 知识库是否必须离线（本地 embedding）？
3. 默认浏览器固定为 Chrome/Edge 之一吗？
4. Live2D 上传是否只接受 ZIP？
5. 人设编辑是否需要“导出为课程模板”？
6. X-ray 是展示“本地源码”还是“GitHub 源码+讲解”？

---

## 10. 推荐落地顺序（最稳）

1. 先做需求 1（启动/全屏）+ 3（背景上传）
2. 再做需求 2（TTS）+ 4（人设）+ 6（Live2D 上传）
3. 最后做需求 5（知识库）+ 7（X-ray 教学入口）

这个顺序能最快让学生“用起来”，同时把高风险模块（知识库）后置，避免拖慢整体交付。
