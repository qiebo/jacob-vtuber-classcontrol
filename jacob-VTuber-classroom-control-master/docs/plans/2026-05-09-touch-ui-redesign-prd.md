# 教学数智人触控界面重设计 PRD

更新时间：2026-05-09

## 1. 背景

当前项目是运行在树莓派 5 上的教学数智人应用，主要以竖屏触控设备展示虚拟人物、背景、字幕与底部交互控件。现有界面在桌面浏览器环境下可用，但在触摸屏上存在明显问题：设置面板字体偏小、可点击区域偏窄、列表选择不直观，老师或学生用手指操作时容易误触。

本次需求聚焦软件界面的重新设计，优先处理左右两侧展开后的设置与字幕体验，不改变底层对话、语音识别、TTS、知识库和人物资源管理能力。

## 2. 产品目标

1. 提升触控可用性：主要控件、列表项、卡片、按钮都要适合手指点击。
2. 降低老师配置成本：左侧设置分类改成更符合教学场景的中文命名。
3. 提升资源选择效率：人物形象和声音从下拉/小列表改为可视化卡片选择。
4. 提升交互反馈感：用户说话时在字幕区域展示声波模拟效果，让系统正在听的状态更直观。
5. 统一视觉风格：右侧字幕/对话区域参考左侧设置面板的风格，形成一致、清晰、美观的教学终端界面。

## 3. 目标用户

- 一线老师：课堂上快速选择人物、声音、知识库和配置项。
- 学生：通过触控屏与数智人交互，能清楚看到系统是否正在听、是否正在回答。
- 部署/运维人员：在树莓派 5 设备上配置默认人物、声音和显示效果。

## 4. 使用环境与约束

- 设备：树莓派 5，外接竖屏触控显示器。
- 操作方式：手指触控为主，鼠标键盘为辅；按单指触控设计，不依赖双指缩放、双指滚动或多点触控手势。
- 展示形态：全屏或 kiosk 模式。
- 当前前端源码入口：`frontend-source/src/renderer/src`。
- 当前设置页相关位置：

## 7. 功能需求

### 7.1 左侧设置整体触控优化

需求描述：

- 设置面板进入后，顶部 tab 文字更大，间距更宽。
- 所有主要可点击控件最小高度不低于 52px，推荐 56px。
- 一级 tab 不应挤成小字横排；当宽度不足时可横向滑动，但当前选中项必须明显。
- 面板内字段减少密集表单感，优先使用卡片、分组、开关、滑杆和大按钮。
- 保存、取消按钮需要保持易点击，避免贴边过小。
- 页面、列表、tab 和滑杆需要支持单指触控操作：上下滑动滚动、横向滑动切换可见内容、单指拖动滑杆。

验收标准：

- 竖屏触控设备上，成年人手指可稳定点击 tab、卡片、按钮。
- 主字号不低于 17px，核心选择项建议 18-20px。
- tab 命名和文案与新信息架构一致。
- “交互”和“其他”不再作为独立 tab 出现。

### 7.2 舞台页人物形象卡片化

需求描述：

- 舞台页中所有人物形象以卡片形式展示，点击卡片即可选择。
- 人物形象包括现有的帧序列形象 AvatarPack 和 Live2D 形象。
- 卡片需要显示：
  - 人物预览图或缩略图
  - 人物名称
  - 类型标签：帧序列 / Live2D
  - 当前使用状态
  - 可选的自定义标签
- 当前选中的人物卡片需要有高亮边框、选中标识或“使用中”状态。
- 上传入口保留，但不要压过选择体验；建议作为人物卡片网格后的独立操作卡片。

交互规则：

- 点击人物卡片后立即切换并显示选中态。
- 删除自定义人物时，需要避免误触；删除按钮应放在卡片次级操作区，并需要确认。
- 如果某个人物缺少缩略图，使用统一占位图或首字母/图标占位。
- AvatarPack 与 Live2D 可以分段展示，也可以在同一网格中用类型标签区分；推荐同一网格展示，减少老师理解成本。

验收标准：

- 至少 2 列卡片布局适配竖屏设置面板。
- 每张卡片高度建议 132-168px，点击区域覆盖整张卡片。
- 切换后主舞台人物更新，卡片出现“使用中”状态。
- 上传新人物后，新人物出现在卡片列表中，并可被选中。

### 7.3 声音页音色卡片化

需求描述：

- 声音页中可选声音以卡片形式展示，替代主要依赖下拉框和手动输入的方式。
- 声音卡片必须支持不同 TTS 供应商动态切换：切换供应商后，卡片列表刷新为该供应商的音色集合，但 UI 结构保持一致。
- 声音卡片需要显示：
  - 声音昵称，如“晓晓”“云希”“亲切女声”
  - 声音 ID 或来源引擎的次级信息
  - 类型标签，如女声、男声、儿童、方言、推荐
  - 当前使用状态
  - 可选试听按钮
- TTS 引擎选择仍然保留，但应放在声音页顶部，作为筛选/分组条件。
- 语速配置建议从文本输入改为滑杆或步进控件，适合触控调整。

交互规则：

- 点击声音卡片后选中声音。
- 每个 TTS 供应商需要记住上次选中的声音；从阿里云切到火山时展示火山音色，从火山切回阿里云时恢复阿里云上次选择。
- 不同供应商字段差异由适配层处理，普通老师只看到统一的“声音卡片 + 语速”界面。
- 点击试听按钮播放一句短测试语音，试听不应改变当前配置，除非同时点击卡片选择。
- 语速滑杆支持默认值复位。
- 高级 JSON 参数可以保留，但默认折叠到“高级设置”，避免老师误操作。

验收标准：

- 常用音色可通过卡片一眼识别并选择。
- 声音卡片点击区域不小于 56px 高。
- 语速调整无需输入复杂参数。
- 高级参数不影响普通老师完成声音选择。

### 7.4 配置页合并交互与其他

需求描述：

- 将原“交互”和“其他”合并为“配置”。
- 配置页承载偏系统和行为的设置项，例如：
  - 麦克风自动开启/关闭
  - 语音识别阈值
  - 验证帧数
  - 是否显示字幕
  - WebSocket 地址
  - 基础 URL
  - 图片压缩质量
  - 截图/图像最大宽度
  - 是否显示群组管理入口
  - 是否显示模式切换入口
- 配置页需要分组展示，不应把所有字段平铺。

推荐分组：

1. 语音交互
2. 显示与字幕
3. 连接与服务
4. 图像与性能
5. 入口显示

验收标准：

- 原“交互”和“其他”的能力在“配置”页中仍可找到。
- 配置页分组清晰，单屏不会出现过密的字段堆叠。

### 7.5 字幕区域声波模拟效果

需求描述：

- 在字幕区域左上角新增声波模拟组件。
- 当用户说话或系统确认正在聆听时，声波开始动画。
- 用户停止说话后，声波停止或切换为低频待机状态。
- 声波要与字幕卡片风格统一，不遮挡字幕文本。

状态规则：

- `listening`：显示活跃声波动画。
- `thinking-speaking`：声波隐藏或转为 AI 回答状态，不表现为用户正在说话。
- `idle` / `waiting` / `loading`：隐藏声波或显示静态小状态点。
- VAD 误触发时，不应持续显示活跃声波。

MVP 实现建议：

- 使用 `AiStateContext.isListening` 控制声波显示。
- 使用 CSS keyframes 制作 5-7 根竖向 bar 的音量起伏效果。
- 组件放在字幕容器左上角，使用绝对定位。

增强实现建议：

- 后续可读取 VAD `previousTriggeredProbability` 或音频帧 RMS，让声波高度与真实音量相关。

验收标准：

- 用户开始说话时，字幕区左上角能在 300ms 内出现声波动画。
- 用户停止说话并进入思考/回答后，声波不再表现为用户输入。
- 声波不遮挡字幕内容，不造成布局跳动。

### 7.6 右侧字幕区域视觉重设计

需求描述：

- 字幕区域需要参考左侧设置面板的深色半透明卡片风格。
- 字体需要比当前更适合远距离阅读和触摸屏观看。
- 用户与 AI 的消息区分要清楚，但不要过度花哨。
- 字幕容器需要有稳定宽度和最大高度，避免长文本撑破舞台。

设计要求：

- 字幕卡片采用半透明深色背景、轻边框、柔和阴影。
- 字幕文字建议 18-20px，行高 1.5-1.65。
- 用户消息和 AI 消息用轻微色彩差异区分。
- 长字幕允许滚动，但滚动条不应太醒目。
- 字幕区与底部输入栏保持安全距离。

验收标准：

- 在竖屏设备上，字幕可读性明显提升。
- 长句换行自然，不遮挡底部输入控件。
- 声波组件和字幕内容视觉上属于同一个模块。

## 8. 视觉与交互原则

1. 触控优先：按钮、tab、卡片都按手指点击设计。
2. 卡片优先：涉及选择的内容优先使用卡片，不优先使用下拉框。
3. 减少专业术语：面向老师的普通设置放前面，高级参数折叠。
4. 状态明确：使用中、上传中、加载中、失败都要明确反馈。
5. 稳定布局：卡片选中、hover、loading 不应导致整体跳动。
6. 树莓派友好：动画轻量，避免高成本 canvas 或复杂粒子效果。
7. 单指可用：所有核心操作必须用单指完成，不把多点触控作为必要路径。

## 10. 数据与接口影响

可能复用的能力：

- 声音试听接口：如果当前后端无试听能力，本期可作为 P1，不阻塞卡片选择。

## 14. 风险与对策

1. 树莓派性能有限
   - 对策：声波使用 CSS 动画，不使用复杂 canvas 粒子。

2. 现有人物资源缺少缩略图
   - 对策：先使用统一占位图；后续补充缩略图生成或上传。

3. TTS 引擎参数差异大
   - 对策：普通模式只暴露“声音”和“语速”；高级 JSON 折叠保留。

4. 设置项合并后老师找不到
   - 对策：配置页按场景分组，并用清晰标题承接原功能。

5. 字幕区域和底部输入栏冲突
   - 对策：定义字幕最大高度和安全间距，长文本内部滚动。

## 15. 待确认问题

1. 声音卡片是否需要第一期就支持试听，还是先支持选择与应用？
2. 人物形象是否需要把 AvatarPack 和 Live2D 混在同一个网格展示，还是分成两个分组？
3. 字幕声波只在用户说话时显示，还是待机时也保留一个小型静态麦克风/状态点？

已确认：

- “任务卡片”指人物卡片/角色卡片，后续统一按人物卡片设计。
- 声音卡片需要覆盖不同 TTS 供应商，通过统一卡片 UI 和供应商适配层实现。
- 树莓派 Ubuntu 触屏按单指触控设计，支持单指滑动、滚动和滑杆拖动，不依赖多点触控。
