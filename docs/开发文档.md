## 1. 总体技术决策（已定型）

| 决策项 | 结论 | 依据 |
|--------|------|------|
| 学生端数据模型 | **方案 A**：用户名作主键 + 班级属性化 | 班级迁移零文件搬迁，主键稳定 |
| 教师端桌面形态 | **pywebview**（系统 WebView2） | 纯 Python 栈、低难度、Win10 1903+ 预装 |
| 用户名规则 | 设备名，仅 `[A-Za-z0-9]`，1–32 字符 | 用户拍板 |
| 形象打包 | 本版全量打包 | 用户拍板，后续版本再加智能判定 |
| 旧档迁移 | 不迁移，全新开始 | 用户拍板 |
| 缩略图频率 | 30 秒，与状态轮询解耦 | 用户拍板 |

### 技术栈

| 层 | 技术 |
|----|------|
| 学生端后端 | Python 3.10–3.12、FastAPI、uvicorn（沿用现有） |
| 学生端前端 | 现有 Vue/React（`frontend-source` 子模块），新增登录页 |
| 教师端后端 | Python、FastAPI（沿用 `teacher-console`） |
| 教师端窗口 | **pywebview**（替换 `webbrowser.open`） |

### 4.4 鉴权与安全

- **教师端 → 学生端**：沿用 `X-Classroom-Token`（来自 `.classroom.env`）。
- **学生端 → 教师机**：新增 `X-Device-Token`（设备令牌，同 `JACOB_CLASSROOM_TOKEN`）。
- **教师端 Web → 教师后端**：登录后下发 JWT/Session Token，前端存 localStorage，后续请求带 `Authorization: Bearer`。
- **学生端 Web → 学生后端**：登录后下发 `session_token`（存 sessionStorage），后续 API 校验。
- **离线降级**：学生端 `check-username` 在教师机超时（默认 2 秒）后走本地注册表，标记 `pending_sync`。
- **本机回环豁免**：`127.0.0.1` 请求免令牌（沿用现有 `require_classroom_token`）。

## 5. 关键算法设计

### 5.1 用户名唯一性仲裁

```
学生端 check-username(username)
  │
  ├─ 1. 格式校验: ^[A-Za-z0-9]{1,32}$
  │     不通过 → 拒绝
  │
  ├─ 2. 本地注册表查询
  │     命中 → 拒绝 "本地已存在"
  │
  ├─ 3. 请求教师机 /api/users/check (超时 2s)
  │     ├─ 在线 + 可用 → 通过
  │     ├─ 在线 + 冲突 → 拒绝 "已被其他设备占用"
  │     └─ 超时/离线 → 进入步骤 4
  │
  └─ 4. 离线降级: 本地注册表无冲突 → 允许创建(pending_sync=true)
        教师机恢复后自动上报 /api/users/sync
```

### 5.2 离线创建同步与冲突处理

```
SyncManager 后台循环 (每 30s 检查一次)
  │
  ├─ 教师机可达?
  │   否 → 继续等待
  │   是 → 遍历本地 pending_sync 用户
  │         │
  │         ├─ POST /api/users/sync {username, device_id}
  │         │     ├─ 教师机无冲突 → 标记 synced
  │         │     └─ 教师机冲突 → 提示用户改名
  │         │           ├─ 改名成功 → 重命名本地目录 + 数据迁移
  │         │           └─ 用户取消 → 保留 pending_sync, 下次再试
  │         │
  │         └─ 重复直到全部处理或教师机再次离线
```

**目录重命名与数据迁移**（改名场景）：

```python
def rename_user(old_username, new_username):
    old_dir = PROFILE_ROOT / old_username
    new_dir = PROFILE_ROOT / new_username
    # 1. 物理移动目录
    old_dir.rename(new_dir)
    # 2. 更新 profile.yaml 内的 username/conf_uid 字段
    profile = read_profile(new_dir)
    profile.username = new_username
    profile.character_config["conf_uid"] = new_username
    write_profile(profile)
    # 3. 更新本地注册表
    update_local_registry(old_username, new_username, pending_sync=False)
    # 4. 更新运行时状态
    if runtime.current_username == old_username:
        runtime.current_username = new_username
```

### 5.3 进度化作品收取

```
教师端 /api/batch/collect
  │
  ├─ 1. 生成 batch_id, 初始化进度记录 (内存 + 可选持久化)
  │
  ├─ 2. 异步任务组 (并发信号量 = 8, 避免一次性 16 台压垮网络)
  │     每台设备:
  │     ├─ GET /classroom/profile/{username}/export (流式)
  │     ├─ 边下载边写临时文件, 每收到 chunk 更新 bytes
  │     ├─ total_bytes 从 Content-Length 获取 (若无则进度按完成与否计)
  │     ├─ 更新 batch progress[device_id]
  │     └─ 下载完成 → 原子替换为 {username}.zip
  │
  ├─ 3. 前端轮询 GET /api/batch/collect/{batch_id} (每 500ms)
  │     或 SSE: GET /api/batch/collect/{batch_id}/events
  │
  └─ 4. 全部完成 → status=done, 返回最终路径列表
```

### 5.4 全量打包（本期）

```python
def pack_workspace(username) -> bytes:
    """全量打包当前用户工作区为 ZIP"""
    profile_dir = PROFILE_ROOT / username
    buffer = io.BytesIO()
    with zipfile.ZipFile(buffer, "w", compression=zipfile.ZIP_DEFLATED) as zip:
        # 1. 档案元数据
        zip.write(profile_dir / "profile.yaml", "profile.yaml")
        zip.write(profile_dir / "manifest.json", "manifest.json")
        # 2. 角色配置 (含形象/背景/声音/性格)
        for path in (profile_dir / "assets").rglob("*"):
            if path.is_file():
                zip.write(path, path.relative_to(profile_dir))
        # 3. 知识库
        for path in (profile_dir / "assets" / "knowledge").rglob("*"):
            ...
        # 4. 联天记录
        for path in (profile_dir / "chat_history").rglob("*"):
            ...
        # 5. 学生文件
        for path in (profile_dir / "files").rglob("*"):
            ...
        # 6. 快照 (可选)
        for ext in ("jpg", "png", "webp"):
            snap = profile_dir / f"snapshot.{ext}"
            if snap.is_file():
                zip.write(snap, snap.name)
    return buffer.getvalue()
```

## 8. 离线降级完整流程

### 8.1 离线创建

```
1. 学生在登录页点"创建用户名"
2. 学生端 POST /auth/check-username
   - 教师机不可达 (2s 超时)
   - 本地注册表无冲突
   → 返回 { available: true, offline: true }
3. 学生端 POST /auth/create
   - 本地创建 profiles/{username}/
   - registry/local_users.json 加入 { username, pending_sync: true }
4. 用户进入主界面, 顶部提示 "离线创建, 待同步"
5. SyncManager 后台每 30s 探测教师机
   - 可达 → POST /api/users/sync
   - 教师机登记成功 → pending_sync: false, 提示条消失
```

### 8.2 离线创建冲突处理

```
场景: pi-01 和 pi-02 同时离线创建了同名 "GroupA"
教师机恢复后:
  pi-01 先 sync → 教师机登记 "GroupA" (device_hint=pi-01)
  pi-02 后 sync → 教师机发现冲突
    → 返回 { synced: false, reason: "conflict", suggested: "GroupA_pi02" }
  pi-02 收到冲突:
    → 提示学生 "用户名已被 pi-01 占用, 请改名"
    → 学生输入新名 (如 "GroupB")
    → rename_user("GroupA", "GroupB")  // 目录重命名+数据保留
    → 再次 sync 成功
```

### 8.3 教师机离线时的教师操作

- 教师**无法登录**教师端（登录是本地的，不依赖学生机，所以可登录）。
- 教师**无法扫描/管控学生机**（因为扫描是向学生机发请求，但学生机在线时教师机可达；若教师机网络全断，则学生机对教师机也不可达，教师端显示全离线）。
- 学生**仍能正常使用**学生端（离线创建/登录/工作）。

## 10. 错误处理与容错

| 场景 | 处理 |
|------|------|
| 学生端创建时教师机超时 | 2s 超时 → 离线降级；不阻塞用户 |
| 学生端 sync 冲突 | 提示改名；改名失败保留 pending_sync |
| 学生端改名时目录被锁 | 重试 3 次；失败则提示重启 |
| 教师端扫描某台离线 | 该台标记 offline，不影响其他 |
| 教师端收取某台超时 | 单台超时(30s) → 该台标记 failed，继续其他 |
| 敹师端收取中网络断开 | 已下载部分保留为 .part；提示重试 |
| WebView2 缺失 | 首次启动检测 → 揹载安装引导 |
| 并发同名创建（教师机在线） | 教师机为唯一仲裁源；先到先得，后到拒绝 |
| 磁盘空间不足 | 打包/保存前检查；不足提示清理 |
| 会话串号（多标签页） | session_token 校验；不匹配拒绝 |
