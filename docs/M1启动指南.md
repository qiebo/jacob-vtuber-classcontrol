## 二、工作模式说明（给新 agent 的内部指令）

### 2.1 核心原则

**串行执行是骨架，子 agent 调研是加速器。**

- 代码改动、测试运行、验收 —— 全部串行，由你（主 agent）自己完成。
- 大范围影响面调研、代码库广度搜索 —— 派 Explore 子 agent 并行加速。

### 2.2 不要做的事

- ❌ 不要派子 agent 写代码（当前可用子 agent 是 Explore 只读类型，不能改文件）
- ❌ 不要让多个子 agent 同时搜不同维度然后期望它们彼此通信（它们只跟你通信）
- ❌ 不要为了"并行"而并行；只在调研确实能并行且省时间时才用

### 2.3 子 agent 的正确用法

子 agent 适合**单一明确的调研问题**，好的派发示例：

✅ 好的：
- "找出学生端所有引用 profile_id 的位置（含 routes、storage、models、tests）"
- "分析 frontend-source 里前端如何调用 /classroom/profile/* 接口"
- "教师端 student_client.py 和 app.py 里有哪些地方依赖 profile_id"

❌ 不好的：
- "帮我重构 storage.py"（子 agent 不能写代码）
- "全面分析这个项目"（太宽泛，结论稀释）
- "规划 M1 的任务"（这是你的职责）

### 2.4 多个 Explore 子 agent 可以并行

如果 M1 开始前要摸清 3 个维度的影响面，**在一条消息里同时发起 3 个子 agent**，它们会并发执行：

```
我在一条消息里同时发 3 个 Agent 调用：
- Agent A: 搜 profile_id 在学生端 Python 的引用
- Agent B: 搜 profile_id 在前端的引用
- Agent C: 搜 profile_id 在测试里的引用
三路并发，等结果汇总后我一次性改准。
```

## 七、常见问题预案

### Q1：跑测试报 ModuleNotFoundError
```powershell
# 在 jacob-VTuber-classroom-control-master 目录下
$env:PYTHONPATH="src"
python -m pytest tests/test_classroom_storage.py -q
```
教师端测试：
```powershell
$env:PYTHONPATH="teacher-console"
python -m pytest teacher-console\tests\test_teacher_console.py -q
```

### Q2：storage.py 重构后测试大面积失败
- 正常预期。M1 是破坏性改造，测试会同步更新。
- 按"models → storage → routes → 测试"顺序改，每层改完跑该层的测试。

### Q3：发现 profile_id 还在某个角落没改干净
- 用 grep 全局搜索：`findstr /s /i "profile_id" src\ teacher-console\`
- 列出所有残留点，逐一判断是否需要改。

### Q4：前端的 /classroom/profile/* 调用要不要在 M1 改？
- M1 优先把后端数据模型改对。前端调用如果传 profile_id，后端可以**临时兼容**（把 profile_id 当 username 用）。
- 前端的彻底改造放到 M2（学生端登录页 + 工作区 UI）一起做。
- 但 build_status 返回字段（current_profile_id → current_username）会影响教师端前端 app.js，需要在 M1 末尾同步。

### Q5：遇到拿不准的设计决策
- 回到开发文档 §3 数据模型、§4 API 设计查权威定义。
- 仍拿不准，回到用户确认。不要自己拍板影响数据模型的决策。
